# 시스템 가이드

이 문서는 `woodcutter_v4`가 어떤 구조로 동작하는지 초보자도 이해할 수 있게 설명한 안내서입니다.

분석 기준 파일:

- `index.html`
- `js/main-unified.js`
- `js/packer.js`
- `js/costCalculator.js`
- `js/settingsManager.js`
- `js/state.js`

## 1. 시스템 개요

이 프로그램은 "판재 1장에 부품들을 어떻게 배치하면 좋은가?"를 계산해 주는 웹 앱입니다.

한눈에 보는 흐름:

1. 사용자가 `index.html` 화면에서 판재 정보와 부품 목록을 입력합니다.
2. `main-unified.js`가 입력값을 읽고 상태(`state.js`)에 반영합니다.
3. 계산 버튼을 누르면 `main-unified.js`가 전단여부, 나무결, 회전 가능 여부를 반영해 계산용 부품 데이터로 바꿉니다.
4. `packer.js`가 실제 배치 계산을 수행합니다.
5. 계산 결과는 다시 상태에 저장됩니다.
6. `costCalculator.js`가 재단 횟수, 총 재단비, 사용률을 계산합니다.
7. `main-unified.js`가 도면, 결과 요약, 부품 현황을 화면에 표시합니다.
8. `settingsManager.js`는 사용자가 바꾼 설정을 저장하고, 다음 접속 때 다시 불러옵니다.

텍스트 흐름도:

`사용자 입력 -> 상태 반영 -> 계산용 데이터 변환 -> 배치 알고리즘 실행 -> 원가 계산 -> 도면/결과 표시 -> 설정 저장`

## 2. UI 구성 설명

화면은 크게 3개 영역으로 나뉩니다.

### 왼쪽: 입력 영역

#### 판재 정보

- `폭 (mm)`: 판재의 짧은 변 또는 가로 길이입니다.
- `길이 (mm)`: 판재의 긴 변 또는 세로 길이입니다.
- `두께 (mm)`: 재단 단가 계산에 사용됩니다.
- `나무결 방향 고려`: 켜면 부품을 자유롭게 돌리지 않고, 긴 방향과 짧은 방향을 맞춰 배치하려고 합니다.
- `전단여부`: 켜면 판재 한쪽에 전단 여백을 두고 계산합니다.

#### 부품 목록

- `치수1`, `치수2`: 부품의 두 변 길이입니다.
- `수량`: 같은 부품이 몇 개인지 입력합니다.
- `회전 가능`: 계산할 때 부품을 90도 돌려도 되는지 정합니다.
- `부품 추가`: 입력한 부품을 목록에 넣습니다.
- `전체 삭제`: 현재 부품 목록을 모두 지웁니다.

#### 계산 버튼

- `최적화 계산`: 현재 입력값을 기준으로 배치 계산을 시작합니다.

### 가운데: 도면 영역

- 계산 전에는 빈 안내 화면이 보입니다.
- 계산 후에는 판재별 배치 도면이 생성됩니다.
- `미리보기`, `PDF 다운로드` 버튼으로 결과를 출력할 수 있습니다.

### 오른쪽: 결과 영역

- `총 판재수`: 필요한 판재 장수
- `재단 횟수`: 계산된 컷 수
- `총 재단비`: 재단 비용 합계
- `부품 배치 현황`: 어떤 부품이 몇 장 배치됐는지 요약

## 3. 핵심 로직 설명

### 3-1. 판재와 부품 개념

이 시스템에서 가장 중요한 개념은 두 가지입니다.

- `판재`: 재단 전의 원본 자재 1장
- `부품`: 판재에서 잘라내야 하는 최종 조각

예를 들어 1220 x 2440 판재 1장에서 여러 개의 부품을 뽑아야 할 때, 시스템은 "어떤 순서와 방향으로 잘라야 가장 효율적인가"를 계산합니다.

계산 전에는 먼저 다음을 확인합니다.

- 판재 크기가 올바른지
- 부품이 1개 이상 있는지
- 부품이 판재보다 명백히 큰지

부품 수량은 계산 전에 낱개로 펼쳐집니다. 예를 들어 같은 부품 5개면 내부 계산에서는 5개의 독립 부품으로 처리합니다.

### 3-2. 나무결 방향 로직

`나무결 방향 고려`를 켜면 부품 회전 자유도가 제한됩니다.

쉽게 말하면:

- 나무결을 고려하지 않으면 부품은 필요에 따라 돌려서 더 잘 들어가게 할 수 있습니다.
- 나무결을 고려하면 부품의 긴 방향과 짧은 방향을 판재의 방향에 맞춰 배치하려고 합니다.

현재 메인 로직에서는 다음 순서로 처리합니다.

1. 판재의 실효 폭/길이를 계산합니다.
2. 그 판재가 어느 쪽이 긴 축인지 판단합니다.
3. 나무결 고려가 켜져 있으면 부품의 긴 변을 판재의 긴 축에 맞추고, 짧은 변을 짧은 축에 맞춥니다.
4. 이 경우 부품 회전은 사실상 제한됩니다.

즉, 나무결 옵션은 단순 표시 기능이 아니라 계산용 치수 자체를 바꾸는 옵션입니다.

### 3-3. 전단여부 로직

`전단여부`는 판재 한쪽에 여백을 두고 계산할지 결정하는 옵션입니다.

현재 로직 기준:

- 전단여부는 기본적으로 항상 `true`가 되도록 설정되어 있습니다.
- 전단 여백 값은 `trimMargin`으로 관리됩니다.
- 실제 계산에서는 판재 `폭`에서 전단 여백을 빼고 사용합니다.

즉:

- 원판 치수: 화면에 표시되는 실제 판재 크기
- 계산 치수: 전단 여백이 빠진 실사용 가능 크기

예시:

- 원판이 `1220 x 2440`
- 전단 여백이 `5`
- 계산용 판재는 `1215 x 2440`

또한 판재 초과 체크는 다음 방식으로 작동합니다.

- 부품의 큰 값은 원판의 큰 값과 비교
- 부품의 작은 값은 전단이 적용된 실제 사용 가능 작은 값과 비교

이렇게 나눈 이유는 "표시용 원판 크기"와 "실제 계산용 크기"를 동시에 반영하기 위해서입니다.

### 3-4. 배치 알고리즘 설명

배치 알고리즘은 `js/packer.js`에 들어 있습니다.

이 파일은 이름이 어렵지만, 쉽게 말하면 "빈 판에 부품을 차곡차곡 넣는 규칙 모음"입니다.

핵심 방식은 `GuillotinePacker`입니다.

이 방식의 특징:

- 칼날이 끝까지 한 번에 지나가는 재단을 기본으로 생각합니다.
- 그래서 실제 재단 공정과 비슷한 형태를 만들기 쉽습니다.

현재 지원 개념:

- `horizontal`: 가로 스트립 우선 배치
- `auto`: 여러 후보를 비교해서 더 나은 방향을 선택

쉽게 설명하면:

1. 먼저 부품들을 하나씩 넣어 봅니다.
2. 어느 방향으로 자르면 공간 낭비가 적은지 비교합니다.
3. 한 판에 다 못 들어가면 다음 판을 새로 만듭니다.
4. 남은 빈 공간이 있으면 추가로 넣어 봅니다.

`auto` 모드는 내부적으로 여러 엔진 후보가 있지만, 현재 기본은 `adaptive` 쪽으로 동작합니다.

이 로직이 하는 일은 크게 세 가지입니다.

- 부품을 판재에 실제 좌표로 배치
- 잘리지 못한 부품 목록 정리
- 절단선 정보(`cutDetails`, `cuttingCount`) 생성

이 절단선 정보는 나중에 도면 표시와 원가 계산에도 사용됩니다.

### 3-5. 원가 계산 로직

원가 계산은 `js/costCalculator.js`에서 담당합니다.

크게 두 가지 방식이 있습니다.

#### 일반 재단비

기본 방식은 아주 단순합니다.

- 판재마다 절단 횟수(`cuttingCount`)를 구합니다.
- 두께에 따라 컷당 단가를 구합니다.
- `절단 횟수 x 컷당 단가`로 재단비를 계산합니다.

두께별 기본 컷당 단가:

- 12T 이하: 1,000원
- 14.5T ~ 23T: 1,500원
- 24T 이상: 2,000원

#### 쭉절단(원판당 고정 단가)

특정 조건을 만족하면 컷당 계산 대신 "원판당 고정 금액"을 적용합니다.

적용 조건은 대략 다음과 같습니다.

- `cutDetails`가 존재해야 함
- 모든 절단이 `fullSpan=true`여야 함
- 실제 잘린 폭 중 90mm 이상이 하나 이상 있어야 함

여기서 실제 폭은 단순 입력값이 아니라 절단 위치 차이(`pos`)를 이용해 다시 계산합니다.

이후 폭을 두 그룹으로 나눕니다.

- `100mm 이하`
- `101mm 이상`

현재 기준은 `Math.round(width)`를 사용합니다. 즉 부동소수점 오차를 줄이기 위해 반올림 후 비교합니다.

판정 방식:

- `count100up`: `Math.round(width) > 100`
- `count100down`: `Math.round(width) <= 100`

100mm 이하 개수가 더 많으면 "100mm 이하 단가"를 씁니다.

두께별 고정 단가:

- 12T 이하: 101mm 이상 5,000원 / 100mm 이하 7,000원
- 14.5~23T: 101mm 이상 7,000원 / 100mm 이하 10,000원
- 24T 이상: 101mm 이상 12,000원 / 100mm 이하 15,000원

다만 고정 단가가 항상 적용되는 것은 아닙니다.

- 고정 단가가 기존 컷당 계산보다 더 비싸면 적용하지 않습니다.
- 이 경우 기존 컷당 단가 계산으로 돌아갑니다.

즉, 시스템은 자동으로 더 유리한 쪽을 선택합니다.

## 4. 주요 변수/파일 역할

### `index.html`

사용자가 직접 보는 화면입니다.

- 입력 폼
- 결과 패널
- 버튼
- 스크립트 로딩 순서

이 파일은 계산을 하지 않고, 계산에 필요한 입력 구조를 제공합니다.

### `js/main-unified.js`

앱의 중심 제어 파일입니다.

주요 역할:

- 이벤트 연결
- 입력값 읽기
- 계산 시작
- 상태 반영
- 도면 그리기
- PDF 저장/미리보기
- 새 프로젝트/불러오기/저장

가장 중요한 함수:

- `init()`: 앱 시작
- `bindEvents()`: 버튼과 입력 이벤트 연결
- `handleAddPart()`: 부품 추가
- `handleCalculate()`: 계산 실행
- `drawBinToCanvas()`: 도면 표시
- `handleNewProject()`: 초기화

### `js/packer.js`

실제 배치 계산 엔진입니다.

주요 역할:

- 부품을 낱개로 펼치기
- 빈 공간에 부품 넣기
- 절단 방향 비교
- 절단선 정보 생성
- 판재가 여러 장 필요할 경우 새 판재 생성

결과로 만들어 주는 값:

- `bins`: 판재별 배치 결과
- `unplaced`: 배치 실패 부품
- `cuttingCount`: 절단 횟수
- `cutDetails`: 절단 위치 상세 정보

### `js/costCalculator.js`

원가 계산 담당입니다.

주요 역할:

- 총 재단비 계산
- 절단 횟수 집계
- 사용률 계산
- 두께별 단가 계산
- 쭉절단 고정 단가 판단

### `js/settingsManager.js`

설정 저장/복원 담당입니다.

주요 역할:

- 기본 설정 제공
- 브라우저 `localStorage` 저장
- 페이지 새로고침 후 설정 복원
- UI에 설정 반영

현재 주의할 점:

- `validateSettings()`에서 `enableTrim`은 항상 `true`로 고정됩니다.
- 즉 사용자가 꺼도 저장 단계에서는 다시 `true`가 됩니다.

### `js/state.js`

앱 전체 상태 저장소입니다.

주요 역할:

- 현재 판재 정보 보관
- 부품 목록 보관
- 설정값 보관
- 계산 결과 보관
- 리스너 호출

쉽게 말하면 앱의 "메모장" 역할입니다.

주요 상태:

- `boardSpec`
- `cuttingList`
- `settings`
- `result`
- `costInfo`
- `labeledGroups`
